# ClickHouse 26.3.17.56 vs 24.3.5.46 (prod) — 2026-07-22

**Verdict: do not upgrade to 26.3 either.** Workload-weighted, 26.3 is **19% slower**
(46.4s → 55.2s) — the same regression profile as [26.6.2.81](26.6.2.81.md), so the ARRAY JOIN
executor regression landed somewhere in **(24.3, 26.3]**. Next bisect step: 25.3 LTS.

## Environment

- Baseline: v24.3.5.46-lts (exact prod version), port 9000; candidate: v26.3.17.56-lts, port 9010
- Both restored from backup `2026-07-21T20-10-54` (26.3 restore of the 24.3-written backup via
  clickhouse-backup 2.8.0 succeeded — backup-compat gate passed)
- Same setup as the 26.6 evaluation (macOS arm64, merges stopped, 45-query corpus, 16 interleaved
  rounds; full results: [26.3.17.56-report.tsv](26.3.17.56-report.tsv))

## Results

- **Correctness**: result rows identical for all 45 queries, zero failures.
- **Regressed 1.2–1.55×**: the same `ARRAY JOIN measures` class as on 26.6
  (300→433 ms, 251→384 ms, 83→113 ms, 22→34 ms); heaviest daily aggregate 1420→1563 ms (+10%).
- **Improved 2–7×**: the same `SELECT DISTINCT` dropdown class (338→47 ms, 212→33 ms).
- **Memory**: heavy aggregations lower, but less dramatically than 26.6 (164→96 MB vs 164→43 MB).

## Diagnosis

Not re-run — the per-query profile matches the 26.6 evaluation (same query class, near-identical
ratios), where the cause was isolated to executor CPU in the ARRAY JOIN path with identical read
volumes ([ClickHouse#91430](https://github.com/ClickHouse/ClickHouse/issues/91430)); neither
`compatibility` nor `enable_analyzer=0` fully mitigates.
