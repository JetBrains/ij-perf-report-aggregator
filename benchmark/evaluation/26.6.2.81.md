# ClickHouse 26.6.2.81 vs 24.3.5.46 (prod) — 2026-07-22

**Verdict: do not upgrade to 26.6.** Workload-weighted, 26.6 is **14% slower** (45.7s → 52.2s
total over the corpus weighted by capture-session frequency). Every `ARRAY JOIN measures`
chart query — the core of each dashboard — regressed 25–55%. Correctness is unaffected.

## Environment

- Baseline: v24.3.5.46-lts (exact prod version), port 9000, `deployment/ch-local/config.xml`
- Candidate: v26.6.2.81-stable, port 9010, `deployment/ch-local-candidate/config.xml`
- Both restored from the same backup `2026-07-21T20-10-54` (restore of the 24.3-written backup
  under 26.6 via clickhouse-backup 2.8.0 succeeded — validates the prod upgrade path)
- macOS arm64, 64 GiB RAM, both servers idle, `SYSTEM STOP MERGES` during the run
- Part layouts near-identical: 2389 vs 2399 active parts, physical rows within 0.006%

## Method

Corpus: `benchmark/corpus.jsonl` — 45 distinct query shapes (34 perfintDev, 10 perfUnitTests,
1 ij) from 760 queries captured in the local query_log while browsing the heavy dashboards.
Runner: `benchmark/run.sh`, 16 rounds interleaved A/B with alternating order, measurements
server-side from each server's query_log, matched via `log_comment`. Full per-query results:
[26.6.2.81-report.tsv](26.6.2.81-report.tsv).

## Results

- **Correctness**: result row counts identical for all 45 queries; zero query failures
  (the 24.3→26.6 analyzer jump breaks nothing in this workload).
- **Regressed 1.2–1.6× (medians)**: all `... ARRAY JOIN measures ...` time-series queries,
  e.g. 299→426 ms, 250→384 ms, 82→106 ms, 21→32 ms. The heaviest daily aggregate:
  1361→1508 ms (+11%).
- **Improved 2–7×**: every `SELECT DISTINCT project|machine ... ORDER BY` dropdown query,
  e.g. 339→49 ms, 224→36 ms, 24→7 ms.
- **Memory**: heavy aggregations use ~75% less (164→43 MB, 119→30 MB peak).

## Diagnosis

Profile comparison on the worst regressions: identical `read_rows`, identical selected marks,
±10% read bytes — the planner reads the same data; the loss is executor CPU in the ARRAY JOIN
path. Matches upstream [ClickHouse#91430](https://github.com/ClickHouse/ClickHouse/issues/91430)
(multiIf + ARRAY JOIN slower under the new analyzer, with large memory savings — the same trade
observed here).

Mitigations tested on 26.6 (9 replays each, top-5 regressed queries):

| query (baseline → 26.6) | `compatibility='24.3'` | `enable_analyzer=0` |
| ----------------------- | ---------------------- | ------------------- |
| 250 → 384 ms            | 392 ms (no effect)     | 292 ms (partial)    |
| 299 → 426 ms            | 403 ms (no effect)     | 326 ms (partial)    |
| 82 → 106 ms             | 102 ms (no effect)     | 96 ms (partial)     |

Neither restores 24.3 performance: this is a within-analyzer executor regression (both versions
run the new analyzer by default), not a settings-default change.
